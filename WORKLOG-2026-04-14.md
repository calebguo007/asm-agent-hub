## 一、产品定位纠偏（最重要的决策）

### 之前的错误理解

把 ASM 前端当成"给人类用的搜索引擎" — 搜索栏、偏好滑块、让人类输入查询。这是**根本性的定位错误**。

### 正确的理解

- ASM 是一个**协议**，产品的用户是 **AI Agent**，不是人类
- 做前端是为了让 **Agent 的主人（人类）看懂** Agent 在做什么决策
- 人类是**观众**，不是操作者
- 偏好设置是 Agent 去问人类的，我们只是告诉 Agent **应该问什么**
- 前端的目的：让人类**信任** Agent 的决策 + 展示生态的**真实性和丰富度**

### 新的设计原则

1. **Built for Agents, not humans** — 这是核心叙事
2. 前端是**监控面板 + 生态全景图**，不是交互界面
3. 数据必须**真实、自信**，不能虚假
4. 像"哄小孩"一样，给人类直观好看的东西

## 十四、前端第四次重写 — "我的 Agent 逛大众点评"视角

### 用户反馈

> "不是看 demo，是要看自家的 agent 怎么逛'大众点评'的"

### 核心视角转变

**旧思路**：展示平台的全景（第三方视角）
**新思路**：展示"我的 Agent"的行为（第一人称视角）

就像你打开大众点评，看到的是"我的订单"、"我的浏览记录"、"我的收藏"，而不是一个静态的餐厅列表。

### 页面结构

1. **My Agent 卡片** — Agent 身份（地址、头像）、余额、决策数、消费金额、使用服务数
2. **Agent Preferences** — 从行为中学习出来的偏好权重（Cost/Quality/Speed/Reliability 四条动态进度条）
3. **Browsing 品类条** — 横向滚动的品类 chip，已浏览的有蓝色高亮，显示 "6/9 categories explored"
4. **Demo Controls** — "Start Agent Journey" + "Next Step" + 进度条
5. **Spending** — 按品类的消费明细（taxonomy → amount）
6. **Activity Timeline** — 核心展示区，时间线格式：
   - 🔗 **Connect** — Agent 连接到 ASM 协议
   - 👀 **Browse** — Agent 浏览某个品类（带服务标签）
   - 🧠 **Decide** — Agent 做决策（意图 → 排名条 → 支付确认）
   - ✅ **Complete** — 旅程结束总结

### 关键设计决策

- **时间线 > 表格**：用时间线展示 Agent 的行为流，不是表格
- **浏览 + 决策交替**：先浏览品类（看了哪些服务），再做决策（比较 + 选择），就像人逛大众点评
- **偏好学习**：从 Agent 的决策权重中反推偏好，动态更新
- **品类高亮**：浏览过的品类在顶部 strip 中高亮，一眼看到覆盖了多少
- **单步模式**：支持 "Next Step" 逐步看，也支持 "Start Journey" 全自动

## 十四、6 大底层问题修复（Session 3）

### 修复清单

| 优先级 | 问题 | 修复方案 | 验证结果 |
|:---:|------|------|------|
| 🔴 P0 | live 模式 txHash 永远 undefined | `res.on("finish")` + 解码 `PAYMENT-RESPONSE` base64 响应头 | 50/50 有 txHash ✅ |
| 🟡 P1 | 没有"调用选中服务"步骤 | Demo 中 ranking 后 `setTimeout()` 模拟调用，输出 actual vs declared 延迟 | 每个场景显示 delta% ✅ |
| 🟡 P1 | Trust jitter 太假 (0.95~1.05) | Box-Muller 正态分布 + 4 维度独立偏差模型 + 异常值概率 | 分数 0.70~0.86 ✅ |
| 🟡 P1 | 缺少 Block Explorer 链接 | Demo Step 7 输出 `https://explorer.arcscan.net/tx/{hash}` | 28 笔链上交易 ✅ |
| 🟢 P2 | 没有 npm 包 | `@asm-protocol/payments`, exports, bin, files, keywords | 完整 npm 配置 ✅ |
| 🟢 P2 | type="commonjs" → ESM | `"type": "module"` + `"module": "Node16"` + `__dirname` polyfill | tsc 零错误 ✅ |

### P0 技术细节

x402/express 中间件的工作流：
```
拦截 → 验证支付 → next() → handler 执行 res.json()
→ 缓冲响应 → settlement → 设置 PAYMENT-RESPONSE 头 → 释放
```

关键发现：`req.payment` 从未被设置。settlement 信息在 `PAYMENT-RESPONSE` 响应头中，base64 编码的 JSON：
```json
{ "success": true, "payer": "0x...", "transaction": "uuid-hash", "network": "eip155:5042002" }
```

解决方案：`res.on("finish")` 在响应完全发送后执行，此时 x402 已设置了响应头。

### P1 Trust 偏差模型

```typescript
DIMENSION_PROFILES = {
  cost:    { mean: 1.00, stddev: 0.02, outlierProb: 0.05, outlierScale: 1.15 },
  quality: { mean: 0.97, stddev: 0.06, outlierProb: 0.10, outlierScale: 0.80 },
  latency: { mean: 1.15, stddev: 0.20, outlierProb: 0.15, outlierScale: 2.50 },
  uptime:  { mean: 0.995, stddev: 0.005, outlierProb: 0.03, outlierScale: 0.85 },
}
```

结果：信任分从 0.97~0.99（全部高且无差异）→ 0.70~0.86（有差异、有层次）
